# 醫院 3D 列印 AI 助手 — 優化方向討論

> 與同事討論用，分為短期（本週可做）、中期（本月）、長期（下階段）

---

## 短期（1-2 天可完成）

### 1. API 回應體驗優化
- **現狀**：流水線完整跑完後，回應訊息很長（合併了全部智能體的輸出）
- **方向**：精簡聊天氣泡中的文字，讓摘要面板承擔展示細節的責任
- **做法**：最終回覆只說「設計包已生成，請查看右側面板」，不重複輸出全部內容

### 2. 錯誤處理強化
- **現狀**：API 呼叫失敗時只有一個通用錯誤提示
- **方向**：區分網路錯誤、API Key 無效、配額耗盡等場景，給出具體指引
- **做法**：在 `llm.ts` 中 catch OpenAI SDK 的各種錯誤碼

### 3. 載入狀態精細化
- **現狀**：只有一個通用的「Processing...」
- **方向**：顯示當前正在執行哪個智能體（「正在分析需求…」→「正在評估風險…」→「正在生成設計摘要…」）
- **做法**：API 回傳的 `nextStep` 已經有了，前端接上即可

---

## 中期（本月內）

### 4. 串流回應 (Streaming)
- **現狀**：使用者需等待整個流水線完成才看到結果（5-15 秒）
- **方向**：使用 Server-Sent Events 或 ReadableStream，即時推送每個階段的輸出
- **影響**：大幅改善感知速度，設計摘要一生成就能看到，不用等材料推薦和工單

### 5. STL 檔案上傳與分析
- **現狀**：上傳按鈕是 disabled 狀態
- **方向**：整合 STL 檔案解析，檢查 printability（壁厚、懸垂、尺寸等）
- **做法**：前端用 Three.js 預覽，後端解析 STL 二進制格式做基本檢查

### 6. 使用者身份區分
- **現狀**：沒有登入系統，任何人都能看到所有項目
- **方向**：最小可行方案 — 簡單的密碼保護或 Hospital ID 輸入
- **做法**：不接入完整 Auth 系統，用 middleware + session cookie 做輕量保護

### 7. 風險評估邏輯改進
- **現狀**：只依賴關鍵字匹配（「surgical」觸發高風險）
- **方向**：讓 LLM 做語意理解，區分「放剪刀的架子」和「手術導板」
- **做法**：將關鍵字匹配降級為輔助信號，以 LLM 判斷為主要依據

## 長期（下階段）

### 8. 列印隊列管理
- 新增「列印中」狀態和列印進度追蹤
- 與實際 3D 印表機管理系統對接（如 OctoPrint API）

### 9. 知識庫 + RAG
- 建立醫院內部材料資料庫、過往案例庫
- 使用 RAG（檢索增強生成）讓 AI 推薦基於歷史數據的材料和設計

### 10. 3D 預覽整合
- 嵌入 Three.js 或 viewer，直接在瀏覽器中預覽生成的 3D 模型
- 支援旋轉、縮放、標註尺寸

### 11. 多語言支援
- 目前只有英文和混合中文的回應
- 根據使用者偏好切換語言

### 12. 通知系統
- 工程師審查完成後通知請求者
- 需求變更時自動推送更新
